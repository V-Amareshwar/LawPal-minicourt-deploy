6. SOFTWARE TESTING

6.1 Introduction
Software testing is a critical phase in the development lifecycle of LawPal-Minicourt to ensure that all integrated components, the React frontend, the Node.js API gateway, and the Python AI backend, function seamlessly and securely. Testing guarantees that the Retrieval-Augmented Generation (RAG) pipeline delivers accurate legal guidance without hallucinations and that user data remains protected. It involves executing the application under various conditions to identify and resolve defects prior to deployment.

6.1.1 Testing Objectives
The primary objectives of testing the LawPal system include:
Validating the correctness of the LLM responses against known legal parameters provided in the ChromaDB vector store.
Ensuring the secure transmission and storage of user authentication data and chat histories.
Confirming that the real-time WebSocket communication operates without significant latency or data loss.
Verifying that the application UI remains responsive and accessible across different browsers and device resolutions.
Testing system resilience by evaluating how it handles API rate limits and external service (Groq API) timeouts.

6.1.2 Testing Strategies
To achieve comprehensive coverage, the following testing strategies are employed:
Unit Testing: Individual modules, such as password hashing functions in Node.js, vector embedding functions in Python, and specific React component rendering, are tested in isolation.
Integration Testing: Verifies the data flow between microservices. A key integration test ensures that user queries sent from React are successfully passed to Node, routed to Flask, matched in ChromaDB, and a valid Groq API response streams back.
Load Testing: Simulates multiple concurrent users connecting via WebSocket to evaluate server performance and ensure the system does not crash under high traffic.
Usability Testing: Conducted to ensure the ChatBot UI and Dashboard are intuitive for users without technical backgrounds.

6.1.3 System Evaluation
System evaluation focuses on the quality of the AI outputs over traditional metrics. The evaluation relies on "ground truth" legal scenarios to ascertain the accuracy of the RAG engine. The system is evaluated on its ability to correctly bypass generic LLM answers in favor of the hyper-localized context injected from the chunked JSON corpus. Speed evaluations are also conducted to ensure the database querying and BM25 ranking process resolve within acceptable user-experience thresholds (under 2-3 seconds).

6.1.4 Testing New System
When updating the corpus with new legal documents or tuning the embedding models, the new system iteration undergoes regression testing. This ensures that previously accurate responses are not degraded by the new vector mappings. The transition involves pointing the Python backend to a localized development ChromaDB, simulating edge-case queries (such as complex bail calculations), and comparing the output logic before pushing the new database state to production.

6.2 Test Cases

Test Case 01: User Authentication
Action: User inputs valid registration credentials and submits. 
Expected Result: Password is encrypted, user is saved in MongoDB, JWT is issued, and redirect to Dashboard occurs. 
Status: Pass

Test Case 02: Unauthorized API Access
Action: User attempts to fetch chat history via REST API without a valid JWT token.
Expected Result: Server intercepts request via middleware and returns a 401 Unauthorized status code.
Status: Pass

Test Case 03: Standard Legal RAG Query
Action: User types "What are the conditions for getting bail?" into the ChatBot.
Expected Result: Python backend performs hybrid search, Groq synthesizes response citing specific constraints from the vector store, and text streams in real-time to the frontend UI.
Status: Pass

Test Case 04: External API Timeout Handling
Action: Simulate an outage or rate limit hit on the external Groq API during a query.
Expected Result: The Node server catches the error and cleanly displays "The AI service is currently unavailable" to the user without crashing the application.
Status: Pass
