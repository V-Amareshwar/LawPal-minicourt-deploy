PROJECT DOCUMENTATION

CHAPTER 1 - INTRODUCTION
1.1 Purpose of the project
The purpose of LawPal-Minicourt is to provide an accessible, AI-driven legal assistant capable of providing automated legal guidance, parsing legal documents, and offering bail calculations using standard legal precedents.

1.2 Problem with Existing Systems
Existing legal systems are often inaccessible to the average user due to high consulting fees, delayed responses, and complex legal jargon that requires professional interpretation. 

1.3 Proposed System
The proposed system is a full-stack web application featuring a React-based frontend for user interaction, a Node.js/Express server for session and user management, and a Python (Flask) backend powering a Retrieval-Augmented Generation (RAG) engine using Groq LLM and ChromaDB.

1.4 Scope of the Project
The scope includes real-time AI chat, vector database integration for semantic legal searches, user authentication, and a scalable architecture to support future expansions in legal modules (e.g., automated bail calculation).

1.5 Architecture Diagram
User Interface (React) <-> API Gateway/Auth (Node.js/MongoDB) <-> AI Inference Engine (Python/Flask + ChromaDB + Groq).


CHAPTER 2 - LITERATURE SURVEY
A thorough study of Natural Language Processing (NLP) in the legal domain, focusing on Retrieval-Augmented Generation (RAG) models. It explores the viability of vector embeddings (Sentence-Transformers) and algorithms like BM25 for accurate legal precedent matching over traditional keyword searches.


CHAPTER 3 - SOFTWARE REQUIREMENT SPECIFICATION
3.1 Introduction to SRS
This section outlines the functional and non-functional requirements necessary to build and deploy LawPal-Minicourt.

3.2 Role of SRS
To establish a clear agreement on system capabilities and technical constraints between developers and stakeholders.

3.3 Requirements Specification Document
A comprehensive list of system inputs, behaviors, and outputs.

3.4 Functional Requirements
- User Registration and Login.
- Real-time chat interface over WebSockets.
- AI query processing using RAG context.
- Storage and retrieval of past chat sessions.

3.5 Non-Functional Requirements
- High availability and low latency response generation.
- Security of user data (password hashing with bcrypt).
- Rate limiting to prevent API abuse.

3.6 Performance Requirements
The system must handle concurrent user requests seamlessly and return LLM responses within a few seconds via streaming or optimized RAG searches.

3.7 Software Requirements
- Frontend: React 19, TypeScript, Vite.
- Backend Servers: Node.js (Express), Python (Flask).
- Databases: MongoDB, ChromaDB (SQLite).

3.8 Hardware Requirements
- Core 2 Duo / i3 Processor or higher.
- 4GB RAM minimum (8GB recommended for vector queries).
- 10GB Free Storage.
